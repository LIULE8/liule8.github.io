性能与SQL优化(1)

性能优化

  • 吞吐与延迟： 有些结论是反直觉的，指导我们关注什么

  • 没有量化就没有改进：监控与度量指标，指导我们怎么去入手

    业务的指标比技术的指标要重要，很多情况下的指标是根据机器来定的，但是可能并不满足业务（用户）的要求。比如订单的指标，突然偏离了正常值，那么一定发生了一些事件，要么是技术的，要么是业务的，对于系统的稳定性有影响。通过对这些异常的追踪，反过来找根本原因。

  • 80/20原则：先优化性能瓶颈问题，指导我们如何去优化

    将系统现在存在的问题排个序，可能前10个问题解决了，系统的性能问题可能就解决了。

  • 过早的优化是万恶之源：指导我们要选择优化的时机

    就像开一间服装工厂，不会一开始就设计一个年产量可以支撑1000亿人穿的工厂，完全不符合需求。所以做一个刚好满足系统初期预计，平均的TPS和QPS，然后峰值 × 5，然后反过来推系统的设计，那么设计就可以简化了。不需要过早的优化性能。

  • 脱离场景谈性能都是耍流氓：指导我们对性能要求要符合实际

性能是一个综合性问题

和功能没有关系的特性，做架构需要考虑的：一致性，稳定性，可维护性

DB/SQL 优化是业务系统性能优化的核心

  1. 业务系统的分类：计算密集型、数据密集型

  2. 业务处理本身无状态，数据状态最终要保存到数据库，所以数据库是拿来兜底的

  3. 一般来说，DB/SQL 操作的消耗在一次处理中占比最大

    一次请求，可能有json序列化，内存做各种判断，最后在去数据库进行保存。比如处理的订单系统，一个订单过来，处理完，再返回结果出去，可能会经过十几次数据库查询，这个就很慢了。假设是用ID的查询精准操作，大概花3-5毫秒，那么这十几次数据库查询就大概是30-50毫秒，所以这整个处理不可能特别快。所以优化一般考虑的是怎么优化数据库。一般操作数据库的就读和写，写一般是要事务，读和写优化了，但是效果不明显，那么就可能是对数据库的理解有问题，数据库DB本身的配置和性能没有调好，或者表结构（数据率设计）没有处理好

  4. 业务系统发展的不同阶段和时期，性能瓶颈要点不同，类似木桶装水

例如传统软件改成 SaaS 软件

以前有些 ERP 的软件是自己带了一个数据库，然后让别人自己可以在店里管理自己的数据，用的好好的，但是这个软件本身的开发和维护的成本很大，C/S 软件每个人的机器不同，用户的 IT 知识匮乏，可能下个新的补丁就启动不了软件了，如果用户机器的某个杀毒软件再把某个文件删了，那么数据可能就恢复不了了。如果改成 SaaS 系统，那么维护成本将急速降低。所有数据都在开发者这里，出了问题由开发者负责恢复。

但是不好改，原来假设有100多个功能，改成多租户的。需要20多个人，半年时间开发，然后再开放给少量用户试用，当试点。看下性能行不行，稳不稳定，功能是否好用，用户体验怎么样。因为原来的C/S的单机系统，对于这些是没有概念，一个小店进销存，一般不会超过1000，100-200是比较常见的，库存也不会超过2000，一般也就700，800。总用户大概1000。那么它一年也就几十万单。数据量非常小，SQL想怎么写就怎么写，表设计几百列都问题不大，一个SQL连十几个表。如果改成SaaS系统，给全国10000个人用，立马完蛋。因为数据量直接增加了10000倍，以前一天100-200单，现在一天直接100-200万单，SQL稍微写的有一点问题，那么直接完蛋。

SaaS 软件，特别是有一定规模的。它在SQL上的问题，跟一般的单机软件完全是两个问题。

补充：

SaaS：软件即服务，功能软件多租户化，从单机版变成web多租户，如之前需要安装金蝶的软件，才可以使用服务，现在只要使用金蝶云就可以使用它的服务了

PaaS：平台即服务，如Google 的GAE，新浪的 SAE。基础设施都是共享的，自己去部署应用

IaaS：Infrastructure as a Service 基础设施即服务，如云平台，openstack，一批资源和硬件，将它多租户划分，然后给多个人使用。看起来你拿到的就是计算机资源，就是一台机器，实际上你拿到是一台虚拟机，是跟别人公用的，自己可以去搞任何东西。

FaaS：函数即服务，如 Serverless，将代码片段丢过去就能 run 起来，是以后的趋势 ===》》headless

DaaS：数据即服务，暂时还不清楚要怎么搞。数据涉及核心竞争力，隐私等问题

什么是关系数据库

1970年 Codd 提出关系模型，以关系代数理论为数学基础 《A Relational Model of Data for Large Shared Data Banks》

一些相关概念：元组(Tuple)、实体、关系，属性

E-R 图

数据库设计范式

  • 第一范式（1NF）：关系 R 属于第一范式，当且仅当R中的每一个属性A的值域只包含原子项
  • 第二范式（2NF）：在满足 1NF 的基础上，消除非主属性对码的部分函数依赖
  • 第三范式（3NF）：在满足 2NF 的基础上，消除非主属性对码的传递函数依赖
  • BC 范式（BCNF）：在满足 3NF 的基础上，消除主属性对码的部分和传递函数依赖
  • 第四范式（4NF）：消除非平凡的多值依赖
  • 第五范式（5NF）：消除一些不合适的连接依赖

1NF：消除重复数据，即每一列都是不可再分的基本数据项； 每个列都是原子的。

2NF：消除部分依赖，表中没有列只与主键的部分相关，即每一行都被主键唯一标识； 每个列都有主键。

3NF：消除传递依赖，消除表中列不依赖主键，而是依赖表中的非主键列的情况，即没 有列是与主键不相关的。从表只引用主表的主键， 即表中每列都和主键相关。

BCNF：Boyce-Codd Normal Form（巴斯-科德范式） 3NF 的基础上消除主属性对于码的部分与传递函数依赖。

SQL 语言

SQL 语言1974年由 Boyce 和 Chamberlin 提出，并首先在 IBM 公司研制的关系数据库系统 SystemR 上实现。

1979年 ORACLE 公司首先提供商用的 SQL，IBM 公司在 DB2和SQL/DS 数据库系统中也实现了 SQL。

1986年10月，美国 ANSI 采用 SQL 作为关系数据库管理系统的标准语言（ANSI X3. 135-1986），后为国际标准化组织（ISO）采纳为国际标准。

1989年，美国 ANSI 采纳在 ANSI X3.135-1989 报告中定义的关系数据库管理系统的 SQL 标准语言，称为 ANSI SQL 89，该标准替代 ANSI X3.135-1986版本。

还有没有其他办法？

结构化查询语言包含 6 个部分：

  1. 数据查询语言（DQL：Data Query Language）：其语句，也称为“数据检索语句”，用以从表 中获得数据，确定数据怎样在应用程序给出。保留字 SELECT 是 DQL（也是所有 SQL）用得最多的动词，其他 DQL 常用的保留字有 WHERE，ORDER BY，GROUP BY 和 HAVING。这些 DQL 保 留字常与其它类型的 SQL 语句一起使用。
  2. 数据操作语言（DML：Data Manipulation Language）：其语句包括动词 INSERT、 UPDATE 和 DELETE。它们分别用于添加、修改和删除。
  3. 事务控制语言（TCL）：它的语句能确保被 DML 语句影响的表的所有行及时得以更新。包括 COMMIT（提交）命令、SAVEPOINT（保存点）命令、ROLLBACK（回滚）命令。
  4. 数据控制语言（DCL）：它的语句通过 GRANT 或 REVOKE 实现权限控制，确定单个用户和用 户组对数据库对象的访问。某些 RDBMS 可用 GRANT 或 REVOKE 控制对表单个列的访问。
  5. 数据定义语言（DDL）：其语句包括动词 CREATE，ALTER 和 DROP。在数据库中创建新表或修改、删除表（CREAT TABLE 或 DROP TABLE）；为表加入索引等。
  6. 指针控制语言（CCL）：它的语句，像 DECLARE CURSOR，FETCH INTO 和 UPDATE WHERE CURRENT 用于对一个或多个表单独行的操作。

MySQL 索引原理

数据是按页来分块的，当一个数据被用 到时，其附近的数据也通常会马上被使用。

InnoDB 使用 B+ 树实现聚集索引

为什么一般单表数据不超过2000万？

因为有一个基础的假设，在MySQL里我们希望每颗B+树层级在2-3层，尽量不要超过3层。超过3层了，如果不再内存里，就得多捞一次，多了一次IO操作。假设我们的ID是BigInt类型，它在数据库里占8个字节，而指针大小在MySQL的 InnoDB 引擎里是6个字节，这样的话一个主键占14个字节， InnoDB 页的大小默认是 16KB，16KB / 14B = 1170。这样的话一个页最大可以放下1170个主键。一颗高度为2的B+树，它可以由16个页组成，所以它能放1170 × 16 = 18720 条数据。假设再加一层，变成3层B+树，则共有1170 × 1170 × 16 = 21902400 条数据。

数据库设计优化

MySQL 数据库设计优化-最佳实践

  1. 如何恰当选择引擎？

    如果都用Innodb也可以，这样就支持行锁和事务了；但是有些场景是不需要事务的，不那么重要，就可以用其他引擎。也可以根据不同表（业务）来选择不同的引擎，在创建表的语句后面指定引擎。

  2. 库表如何命名？

    数据库的规范，创建表，字段的时候怎么规范的命名。有意义的单词，下划线，前缀等

  3. 如何合理拆分宽表？

    一个表列数不能太多。 ===》重构

  4. 如何选择恰当数据类型：明确、尽量小

    • 明确：意味着长度固定。小：
    • char、varchar 的选择
    • （text/blob/clob）的使用问题？只要用了blob/clob，性能立马下降5-10倍。

  5. 文件、图片是否要存入到数据库？不要存，尽量存结构化的数据。而且文件和图片数据比较大，会打破我们对于B+树索引的估算。尽量将这些大文件存到分布式的文件系统上。

  6. 时间日期的存储问题？会有时区问题。可以尝试用long类型存时间，这样就不会有任何时间运算，比较大小直接比就行。然后给前端再经过一个转换器即可。

  7. 数值的精度问题？很多东西的精度超出了int和float的范围。可以乘个多少倍变成整数存起来。数据库可以存精度和值，这样就跳过了数值的精度问题。

  8. 是否使用外键、触发器？绝大多数情况下，都不建议使用外键和触发器。如果不使用外键需要在业务系统里保证数据的关系。因为有外键会多了一步检查操作，例如往订单详情表插入一条订单数据，会用这个外键的订单ID，去订单表查询是否有这个订单，有才允许你插入，没有则抛错。如果有强制性事务保证一致，或者柔性事务保证最终一致，那么就可以将外键去掉。这样程序更简单，插入和修改数据性能更高。

还有哪些常见问题？

  1. 唯一约束和索引的关系？

    唯一约束都会默认创建一个唯一索引

  2. 是否可以冗余字段？

    如果需要，则可以放心大胆的去创建冗余字段，只要能够使你的业务系统能够更高效的允许。

  3. 是否使用游标、变量、视图、自定义函数、存储过程？绝大多数是不允许的。

  4. 自增主键的使用问题？在单表的环境下可以使用，但是在分布式环境下不建议使用

  5. 能够在线修改表结构（DDL 操作）？这种是会将这个表锁住，影响其他业务性能

  6. 逻辑删除还是物理删除？一般重要的数据，建议用逻辑删除

  7. 要不要加 create_time，update_time 时间戳？非常建议加；比如增量的同步数据到别的地方很有用，做数据处理，或者分批查询数据也有用

  9. 如何快速导入导出、备份数据？
