ConcurrentHashMap面试指南

ConcurrentHashMap涉及的知识点：HashMap，HashTable，UnSafe，CAS，数组+链表，Segment，ReentrantLock（非公平锁，公平锁），红黑树。

大多数程序员都是业务开发者，在面试官眼里，如果没有 3 到 5 年的开发经验，未必能围绕底层源码，全面掌握并展示线程和排查 OOM 问题等技能。但如果求职者能够通过 ConcurrentHashMap 这个切入点全面展示技能，那么就能很大程度上提升面试成功的可能性。

接下来，就从底层源码和技术面试这两个角度，全面分析下 ConcurrentHashMap 对象。

首先会通过底层源码，从线程并发角度，分析读写 ConcurrentHashMap 对象的详细做法，然后，在此基础上教你如何在面试中展示这方面的亮点；不仅如此，还会教你在面试中展示“通过分析源码排查 OOM 问题”的说辞。

先看数据结构，其中包含面试必问点 volatile

ConcurrentHashMap 是以键值对的形式存储数据，在 JDK 1.8 及之上的版本里，ConcurrentHashMap 对象会用 node 数组（左图）或红黑树（右图）的方式存储键值对类型的数据，如下图所示。

之所以用两种方式来存储，主要是因为当 ConcurrentHashMap 里存储的键值对出现冲突，且冲突的数量比较多时，用右图红黑树的方式能够提升数据读写的性能。ConcurrentHashMap 用来存储键值对的 Node 对象如下所示，其中包含了面试热点“volatile 关键字”，在下文中会具体分析。

通过 Node 源码可以看到，其中用 key 和 value 存储键值对，如果遇到冲突，则用 next 对象指向下一个具有相同 key 的 Node 对象。这里请你思考，为什么要在 val 和 next 变量前加 volatile ？下面我们具体分析一下。

在多个线程同时写一个 ConcurrentHashMap 对象时，线程内存和主内存的关系如下图所示。

多个线程会先把对象保存在本线程内部的内存中，随后再回写到主内存，这样操作的弊端是，当多线程并发时，就有可能引发数据问题。

而变量一旦被 volatile 修饰，那么多个线程会直接读写主内存中的该变量，这样就避免了引发数据冲突。也就是说，这里通过分析 ConcurrentHashMap 的 Node 源码，引出了多线程面试的一个关键点——volatile 的含义。

通过 ConcurrentHashMap 对象展示你的技能

虽然 ConcurrentHashMap 对象包含了很多“值钱”的技术，但在面试中，建议你不要自说自话地发表看法，需要找机会说出或引导面试官问及。下面我会给出几种方式，供你参考。

  1. 当面试官问，你知不知道 volatile、Synchronized 或 CAS 机制，你便可以顺着他的思路往下说。

    参考思路：我知道，另外我还知道 ConcurrentHashMap 对象中用到这些，然后你可以写出本文中给出的代码并进行讲解。

  2. 当面试官问，你知不知道线程安全机制锁机制，或者线程并发机制。

    参考思路：我知道，而且我通读过 ConcurrentHashMap 对象的源码，从中研究过此类线程机制，然后你可以从 Node 结构讲起，边讲边写源代码或画框图，从读写 ConcurrentHashMap 对象的源码角度，讲述线程并发机制。

  3. 当面试官问及任何 HashMap 或集合对象时。

    参考思路：你在回答好本身问题后，可以再多提一句，HashMap 是线程不安全的。所谓线程不安全，是指多个线程同时写时会出问题，如果遇到这种需求，可以用 ConcurrentHashMap 对象，也可以从 Node 结构讲起，结合读写方法的源码展示技能。

你也可以举一反三，只要面试官问及任何线程、集合或数据结构方面的问题，你都可以把话题牵引到 ConcurrentHashMap 对象上，再按本文给出的步骤，结合源码展示技能。甚至，如果面试官没问及，你也可以用如下的方法主动提及，毕竟 ConcurrentHashMap 对象里包含了很多“值钱”的技能点。

  1. 在介绍项目环节，你可以说，在这个项目里，我们用到了不少集合对象，在单线程场景里，我们用到了 HashMap 等，但如果在多线程场景，同时要确保多个线程能正确读写键值类对象，我们就用到了 ConcurrentHashMap 对象。

  2. 一般面试官在问完问题后也会让求职者提问，此时你可以主动说，平时我还看了不少源码，从中了解到了 ConcurrentHashMap 对象数据结构。当面试官接过话题继续提问时，你可以从 ConcurrentHashMap 的 Node 键值对数据结构讲起，着重提到 volatile 关键字的作用，然后按“读不加锁写加锁”的思路，讲述读写相关的方法，在其中可以突出 Node 数组和红黑树等数据结构。

甚至，ConcurrentHashMap 对象是根据哈希表结构存储键值对数据，以及用桶链路的方式处理冲突，这本身也是一个非常好的展示技能的角度。

面试加分项：性能调优要点

在面试中，既然已经聊到了 ConcurrentHashMap 对象，那么其实你可以再进一步，通过如下的方式展示性能调优方面的高阶技能。

  1. 展示内存性能调优方面的技能

    在项目上线前，我们做了压力测试，从中发现订单等模块的内存用量过大，通过观察日志，我们发现是由于 ConcurrentHashMap 对象保存的数据过多，用完以后没及时 clear 掉，由此引发了 OOM 问题。因此当我们用好数据以后，应该立即 clear 掉该 ConcurrentHashMap 对象，内存性能就会好很多了。

  2. 展示多线程环境下数据读写方面的调优技能

    在项目的数据同步模块中，需要把客户文件缓存到本地读写。一开始我们用的是 HashMap，但 HashMap 不是线程安全的，所以读写时都用了 Synchronized 关键字，这样处理的弊端是，当读写线程个数变多时，由于只有一个线程能读写，因此其他线程会阻塞。不仅性能慢，而且还会导致死锁。

    后来我们改用 ConcurrentHashMap 对象，该对象对读操作不加锁，对写操作时会采用 synchronized+CAS 的并发策略，所以一定程度上提升了读写性能。

通关面试经历

这里就分享一些用 ConcurrentHashMap 方面的说辞，通关技术面试的经历。

经历 1：我去面试一家 200 人规模的创业公司

当面试官提问：你知不知道 volatile 关键字？
我回答：知道，而且我还知道 ConcurrentHashMap 对象的数据结构中，用到了这个关键字。

此时面试官就很感兴趣，于是我就边写 Node 源码，边讲述 volatile 的作用，随后顺带提到了“写 ConcurrentHashMap”对象时的“锁机制”，我甚至还没有提到底层 Node 数组和红黑树，也还没讲到“读 ConcurrentHashMap”的源码，面试官就直接让我通过面试了。

经历 2：我去面试一家上万人规模的互联网公司

面试官问：你熟不熟 Java 内存调优？
我回答：我知道，而且我还解决过因 ConcurrentHashMap 对象没释放而导致的性能问题。

首先提到因没有 clear（清空） ConcurrentHashMap 对象，而导致了 OOM 问题；接着我又说了具体如何解决——无非就 clear 掉；然后结合项目进一步介绍，在项目中由于要确保在多线程并发下读取数据的正确性，所以用到了 ConcurrentHashMap，最后再展开聊了聊读写相关的源码。
接下来面试官就草草再问了些 Java 框架方面问题，就让我通过了面试，整场面试才持续一刻钟。

经历 3：我去面试一下知名外企

面试官问：你熟不熟悉 Synchronized？
我回答：这个关键字是起到了“锁”的作用，在 ConcurrentHashMap 源码中用到这个。

然后我又结合 ConcurrentHashMap 对象的写方法，讲述了 Synchronized 和 CAS，以及 ConcurrentHashMap 数据结构，最终顺利过了面试。

总结

以上就是“ConcurrentHashMap 面试指南”的全部内容，我们不仅通过源码，从多线程并发角度讲述了 ConcurrentHashMap 的数据结构和读写操作方法，还给出了通过 ConcurrentHashMap 对象在面试中展示技能的说辞的方法。

在面试前，你可以参考这里给出的思路，准备一些把回答引导到 ConcurrentHashMap 方面的话术。毕竟这方面包含的技能太值钱，你面试时不展示就很吃亏了。

其实你面试时自己也有体会，大多数情况都是在被动地回答问题，但如果你在面试中，能够不露痕迹地引导话题，通过 ConcurrentHashMap 这个切入点，向面试官全方位地展示数据结构、线程并发和性能调优等方面的技能时，面试官一定会认为你比其他求职者更优秀。
