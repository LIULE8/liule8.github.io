RESTful会被新版GraphQL干掉吗？

Spring 社区正式宣布 Spring GraphQL 成为其家族的顶级项目，并发布了 1.0 里程碑版本。

而此时也正逢 GraphQL Java 诞生 6 周年，可以说 Spring GraphQL 的核心价值便是将 GraphQL Java 集成到了 Spring 生态中。

一直以来，都少不了 GraphQL 与 RESTful 这两种前后端交互 Web API 的对比讨论。那么在新模式下，RESTful 会被这个新版 GraphQL 超越甚至替代吗？会为你接下来的工作、业务带来新变化吗？

想要 GraphQL 落地，就需要采用对应的开发框架。将基于 Spring GraphQL 这款崭新的开发框架，给出精简而又完整的原创性案例代码，并展示其开发流程和实战技巧，而这些代码完全可以直接应用到你的日常开发中。

新版 GraphQL：从 Java 走向 Spring

我们知道，GraphQL 是一种理念和规范，它并不直接提供开发工具和框架。

而 GraphQL Java 是基于 Java 开发的一个 GraphQL 实现库，但这个实现库一直都还只是一个执行 GraphQL 请求的引擎。在实际的应用开发中，开发人员还需要创建自己的 HTTP 适配器来与业务代码完成整合。

在 Java 世界中，Spring 是目前最主流的开发框架，没有之一。而 Spring GraphQL 的诞生，则为使用 Spring 框架的开发人员提供了针对 GraphQL 的一站式开发体验。这将大幅度降低 GraphQL 的开发难度和成本，也将极大促进广大 Spring 框架的开发人员熟悉并掌握 GraphQL，从而推动 GraphQL 落地。

RESTful 会被新版 GraphQL 干掉吗？

新版本诞生后，它能克服哪些困难，又补足哪些空白呢？它的未来潜力又如何呢？下面从还原真实工作场景开始展开讨论。

这时前后端之间就需要重新明确 API 定义，并再一次进行联调了，而这个过程实际上是非常浪费的。

我们每天都可能在反复经历着类似的场景。在这些场景中，前后端开发人员已经意识到：传统的 RESTful API 并不能非常好地满足，前后端分离场景下的交互需求。

我们可以进一步把 RESTful API 存在的问题做一些梳理，它存在以下四个典型问题。

  • 前端无法预判响应的数据格式

    正如上述场景所展示，一旦服务端对数据结构做了任何改变，前端都只能被动接收，而无法在发起请求之前感知这种改变。

  • 无法根据请求控制对应的返回结果

    如在上述场景中，前端请求可能只想获取 User 对象中的“name”和“age”字段，而不需要“address”字段。显然 RESTful API 无法满足这种述求，除非另外开发一个 HTTP 端点。我们知道，数据在网络中的传输是需要成本的，无法按需获取数据，同样导致了不必要的资源浪费。

  • 多次请求问题

    这个问题在现实开发过程中也非常典型。再次回到上述场景中，假设 User 对象中包含了一组家庭成员信息。那么基于 RESTful API，如果想获取这些数据，就只能再发起一个专门的请求来根据 User 的 id 获取对应的家庭成员列表，例如以下的 RESTful API 的多次请求示意图。

  • 请求地址过多问题

    当然我们也可以针对该需求，专门设计一个能够同时返回用户信息和家庭成员信息的接口。但这就又会引出第四个典型问题，即请求地址过多问题。

    如果针对各个具体场景我们都需要一一暴露专门的 HTTP 端点，那么在一个系统中 HTTP 端点数量会非常庞大，难以维护和管理。

好了，RESTful API 的问题已经暴露得非常清楚了。那么如何解决这些问题呢？这就需要引入全新的技术体系——GraphQL。

相比 REST，GraphQL 可以说是一项比较新的技术体系：2012 年诞生在 Facebook 内部，并于 2015 年正式开源。顾名思义，GraphQL 是一种基于图（Graph）的查询语言（Query Language，QL），它从根本上改变了前后端交互 API 的定义和实现方式。

接下来，就让我们一起来看看 GraphQL 是如何解决 RESTful API 所面临的一系列问题的。

GraphQL 是如何解决 RESTful API 面临的问题？

要想使用 GraphQL，我们首先需要关注它发送请求的方式。针对获取用户信息这个场景，一个典型的请求示例如下所示：

可以看到基于 GraphQL 的请求方式与使用 RESTful API 有很大的不同。除了在请求体中指定了目标 User 对象的参数 id 值之外，我们还额外指定了“name”和“age”这两个参数，也就是告诉服务器端这次请求所希望获取的数据字段。

  • 显然，这种请求方式完美解决了 RESTful API 中无法根据请求控制对应返回结果的问题；

  • 同时，这种请求方式也解决了前端无法预判响应的数据格式问题（也就是 RESTful API 存在的四个典型问题中的前两个问题）。

因为前端在请求的同时，已经知道从服务端返回的数据字段就是请求中指定的字段，因此前端就不需要再对响应结果进行专门的判断和处理。

在该请求中，我们一方面指定了想要获取的 User 对象中的“name”和“age”字段，同时也指定了需要获取用户对应的家庭成员列表字段“members”以及它的子字段“name”。这样，通过一次请求，我们就可以同时获取用户信息和家庭成员信息，而不需要像 RESTful API 那样需要发送两次请求。

这里，你可能已经注意到：通过 GraphQL 发起请求实际上只需要指定一个 HTTP 端点地址即可。因为我们可以基于同一个端点通过传入不同的参数而获取不同的结果，也就不需要专门设计一批 HTTP 端点来分别处理不同的请求了。

关于 GraphQL 与 RESTful API 的对比就讲到这里。理解了 RESTful API 所存在的核心问题通过 GraphQL 都可以得到解决。

RESTful 与 GraphQL，到底选哪个？

首先明确，我并不推荐你在任何场景下都使用 GraphQL。

对于那些 API 定义与资源概念匹配度较高、也不需要实现类似“用户信息内部嵌套家庭成员信息”的复杂查询场景，传统的 RESTful API 仍然是首选。它的各个 HTTP 端点之间相互独立，职责非常明确。

但对如下场景，GraphQL 则更有优势。

  • 业务复杂度较高的场景

    因为复杂度的产生很大程度上是由于应用程序关联的系统、模块以及功能之间的联动与交互程度很高。这时候如果采用 RESTful 所提供的基于资源的方式来设计方法，接口定义与资源概念一般很难匹配，也就无法很好表达 API 的含义。

  • 需求变化很快的场景

    对于互联网应用而言，业务和需求变化很快，对应的 API 定义和数据结构同样也是多变的。这时候，通过引入 GraphQL 能够有效降低沟通成本。这点对于分布式办公场景尤其明显，因为在这种场景下前后端同学往往需要反复确认 API 定义和响应数据类型，沟通成本很高。

  • 弱文档化管理的场景

    如果你的团队采用的敏捷开发模式，文档化管理的流程和工具使用比较薄弱的话，同样建议你使用 GraphQL。GraphQL 把开发人员原本用来维护文档的时间释放出来，通过请求和响应结果之间的对应关系，解决前后端数据的一致性和同步性。

RESTful 和 GraphQL 的并存发展

如果你已经实现了一部分 RESTful 服务，那么可以让 GraphQL 与这部分 RESTful API 并存发展。

尤其是对于那些单一的 RESTful 服务，可以把 GraphQL 直接连接到已有的 RESTful 服务上。通过这种策略，RESTful 服务中已经实现的业务逻辑层、数据访问层组件都可以得到复用。
我们要做的只是开放一个新的 GraphQL 访问入口而已。

如果你正在采用微服务架构，那么引入 GraphQL 的策略就可以是在所有后端微服务之前架设一层由 GraphQL 构建的数据层，并对后端服务提供的数据进行自由的组装之后，开放给前端。这种实施策略，类似于微服务架构中的 API 网关，一方面对前端请求进行适配和路由，一方面完成前后端之间的解耦。

“RESTful vs GraphQL”之我见

在前面的讨论中，已经分析了 RESTful 和 GraphQL 的各自特性，也给出了使用 GraphQL 的一些典型场景。虽然这是一篇描述 GraphQL 以及 Spring GraphQL 的文章，但还是需要认识到，在未来很长一段时间内，RESTful 仍然将是开发 Web 应用程序的主流架构风格。

  • RESTful 面向资源的操作语义容易理解和实现，针对超媒体的 HATEOAS 技术也为开发自解释型的 Web API 提供了很好的支持；

  • 而 GraphQL 在服务端系统的设计难易度以及性能等方面存在的问题，在一定程度上阻碍了开发人员使用这项新兴技术。

然而，GraphQL 的成长潜力是巨大的。随着越来越多的开发语言提供了 GraphQL 实现库，传统意义上前后端职责明确分离、各司其职的局面将会打破。无论是前端还是服务端开发人员，都可以基于 GraphQL，实现满足前后端交互所需要的数据层，这种数据层是很轻、很薄的，其实现过程并不会比 RESTful 复杂多少。

同时，随着设计理念的不断演进，除了 Query 和 Mutation 这两种基本操作之外，GraphQL 也计划引入面向订阅的第三个操作符 Subscription，以便直接从服务器推送数据。这种类似响应式编程的技术体系，能够很好地解决传统前后端交互所采用的的“请求-响应式模型”所面临的性能问题。

Spring 家族对 GraphQL 的加持，是又一次对 GraphQL 成长潜力的加码。那么你知道 Spring 对 GraphQL 提供了哪些具体的解决方案？刚成为 Spring 顶级项目的 Spring GraphQL 框架又具备哪些功能特性吗？让我们一起来看一下。

争议话题讨论：实现 GraphQL，前端驱动？服务端驱动？

对于大多数系统而言，前后端都需要紧密的配合。在这个配合过程中，前端的痛点往往是大于服务端的。所以，一般场景下前端对于引入 GraphQL 的诉求要大于服务端。

服务端同学

如果采用和 RESTful API 一样的开发模式，那么实现 GraphQL 的工作量主要是在服务端。服务端同学需要基于 GraphQL 规范，重新设计并实现 API。如下图所示，通常都建议单独构建一个数据层来对外暴露 GraphQL API。

前端同学

请注意：GraphQL 只是一种规范，并不是开发语言。而支持 GraphQL 的开发语言有很多种，既包括面向前端的 Javascript，也包括面向服务端的 Java、GO 等。所以原则上，前端和服务端都有能力去实现 GraphQL。
