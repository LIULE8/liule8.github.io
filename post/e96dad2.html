从 Spring 到 Spring Boot

Spring 变得越来越复杂

配置的发展方向：

  • XML–全局
  • 注解–类
  • 配置类–方法

Spring 4 以上的新特性，走向 Spring Boot

功能，使用方式太复杂，怎么办？

Spring Boot 的出发点

Spring 臃肿以后的必然选择，一切都是为了简化

  • 让开发变简单

  • 让配置变简单

  • 让运行变简单

怎么变简单？关键词：整合

就像是 SSH、SSM，国产的 SpringSide

基于什么变简单：约定大于配置。

限定性框架和非限定性框架？

  • 非限定性框架就是哪里都能用，但是当它哪里都能用 之后的时候，那它一定不是最好的，就像一个人什么技术都会，那他会的技术一定不是最牛逼的。
  • 限定性框架就是给你限定死了一些场景或者配置，只适合某些场景。

Spring Boot 如何做到简化

为什么能做到简化：

  1. Spring 本身技术的成熟与完善，各方面第三方组件的成熟集成
  2. Spring 团队在去 web 容器化等方面的努力
  3. 基于 MAVEN 与 POM 的 Java 生态体系，整合 POM 模板成为可能
  4. 避免大量 maven 导入和各种版本冲突

Spring Boot 是 Spring 的一套快速配置脚手架，关注于自动配置，配置驱动

什么是框架，脚手架，解决方案？

  • Spring -> 框架：就是跟很多工具整合好的，跟业务无关的东西
  • Spring Boot -> 脚手架：整合了各种框架，解决好冲突，可以直接拿过来简单的改下，就可以进行业务开发
  • Spring Cloud -> 微服务的解决方案：解决方案是针对某一领域的一整套的解决办法

什么是 Spring Boot

Spring Boot 使创建独立运行、生产级别的 Spring 应用变得容易，你可以直接运行它。 我们对 Spring 平台和第三方库采用限定性视角，以此让大家能以最小的成本上手。 大部分 Spring Boot 应用仅仅需要最少量的配置。

功能特性：

  1. 创建独立运行的 Spring 应用
  2. 直接嵌入 Tomcat 或 Jetty，Undertow，无需部署 WAR 包
  3. 提供限定性的 starter 依赖简化配置（就是脚手架）
  4. 在必要时自动化配置 Spring 和其他三方依赖库
  5. 提供生产 production-ready 特性，例如指标度量，健康检查，外部配置等
  6. 完全零代码生产和不需要 XML 配置

如果不用Spring Boot打包好的jar包，也可以用Maven进行打包，用Maven官方的工具Shade，会把一整个类库打包好。

Spring Boot 核心原理*

Spring Boot 两大核心原理

  1. 自动化配置：简化配置核心基于 Configuration，EnableXX，Condition

  2. spring-boot-starter：脚手架核心整合各种第三方类库，协同工具

application.yml Configuration Bean
前缀 一组配置 Starter 组件

为什么要约定大于配置

举例来说，JVM 有1000多个参数，但是我们不需要一个参数，就能 java Hello

优势在于，开箱即用：

  1. Maven 的目录结构：默认有 resources 文件夹存放配置文件。默认打包方式为 jar
  2. 默认的配置文件：application.properties 或 application.yml 文件
  3. 默认通过 spring.profiles.active 属性来决定运行环境时的配置文件
  4. EnableAutoConfiguration 默认对于依赖的 starter 进行自动装载
  5. spring-boot-start-web 中默认包含 spring-mvc 相关依赖以及内置的 web容器，使得构建一个 web 应用更加简单

自动化配置原理

Spring Boot 自动配置注解

@SpringBootApplication： SpringBoot 应用标注在某个类上说明这个类是 SpringBoot 的主配置类，SpringBoot 就会运行 这个类的 main 方法来启动 SpringBoot 项目

核心启动入口

  • @SpringBootConfiguration 加载 SpringBoot 配置
  • @EnableAutoConfiguration
  • @AutoConfigurationPackage
  • @Import({AutoConfigurationImportSelector.class}) selector 会把适合当前 application 的配置加载处理

加载所有 META-INF/spring.factories 中存在的配置类（类似 SpringMVC 中加载所有 converter）

条件化自动配置

运行时灵活组装，避免冲突

  • @ConditionalOnBean 判断某个Bean存在才配置
  • @ConditionalOnClass 判断有某个类存在才配置：如动态调整算法之类的
  • @ConditionalOnMissingBean 判断某个Bean不存在才配置：如DataSource，事务管理器等只能存在一个对象
  • @ConditionalOnProperty
  • @ConditionalOnResource
  • @ConditionalOnSingleCandidate
  • @ConditionalOnWebApplication

JDBC 与数据库连接池*

JDBC 是 Java 里操作数据库的核心

JDBC 定义了数据库交互接口：

  • DriverManager
  • Connection
  • Statement
  • ResultSet
  • 后来又加了DataSource–Pool

Java 操作数据库的各种类库，都可以看做是在 JDBC 上做的增强实现

为什么可以这么做？

加上 XA 事务–XAConnection

从连接池获取–PooledConnection

MySQL 驱动 JDBC 接口–Connection

数据库连接池

  • C3P0
  • DBCP – Apache CommonPool
  • Druid – 德鲁伊，主要通过AOP思想，代理connection，在execute方法前后加上自定义代码，做增强。
  • Hikari – 日本人写的数据库连接池

Hikari 是最快的，Druid是最重的，如果只是监控没必要用，可以用其他APM工具

而且这些连接池的快慢意义不是特别大，它们都很快，区别可能是启动那一下的速度的区别，启动后的使用基本没有区别。

同理 JSON 序列化，相差都不是特别大

假设一个接口，从获取connection（零点几毫秒），到查询数据库（100毫秒），到序列化（零点几毫秒）返回结果

序列化你可以从零点几毫秒优化到零点零几毫秒，对于整体性能影响不大（量太小没有意义）。

而查询数据库，如果从100毫秒。优化到10毫秒以内，对于整体性能就能提升10倍。

JSON库类比，fastjson是最快的，但是他有一个缺点，就是他对json标准支持不好，导致了spring默认没有用fastjson。spring支持了jackson1和jackson2。

fastjson 对于一些错误格式的 json串也可以转格式。老外比较认标准。而且还改不了，因为阿里内部已经大量使用了fastjson了，如果现在把他的行为改了，那么已经使用的fastjson的那些系统都跑不了了。fastjson现在有5000多个测试用例，里面包含很多这种错误格式json的用例

ORM-Hibernate/MyBatis*

Hibernate

ORM（Object-Relational Mapping） 表示对象关系映射。Hibernate 是一个开源的对象关系映射框架，它对 JDBC 进行了非常轻量级的对象封装，它将 POJO 与 数据库表建立映射关系，是一个全自动的 orm 框架 ，hibernate 可以自动生成 SQL 语句，自动执行， 使得 Java 程序员可以使用面向对象的思维来操纵数 据库。

Hibernate 里需要定义实体类和 hbm 映射关系文件 （IDE 一般有工具生成）

Hibernate 里可以使用 HQL、Criteria、Native SQL 三种方式操作数据库，也可以作为 JPA 适配实现，使用 JPA 接口操作。

Hibernate 可以根据映射文件将 HQL 的语句转换为普通的 SQL 语句

MyBatis

MyBatis 是一款优秀的持久层框架， 它支持定制化 SQL、存储过程以及高 级映射。MyBatis 避免了几乎所有的 JDBC 代码和手动设置参数以及获取结 果集。MyBatis 可以使用简单的 XML 或注解来配置和映射原生信息，将接 口和 Java 的 POJOs(Plain Old Java Objects,普通的 Java 对象)映射成数 据库中的记录。

MyBatis-半自动化 ORM

  1. 需要使用映射文件 mapper.xml 定义 map 规则和 SQL
  2. 需要定义 mapper/DAO，基于 xml 规则，操作数据库

可以使用工具生成基础的 mapper.xml 和 mapper/DAO

一个经验就是，继承生成的 mapper，而不是覆盖掉（这样不会影响旧的内容），也可以直接在 mapper 上用注解方式配置 SQL

MyBatis 与 Hibernate 比较

MyBatis 与 Hibernate 的区别与联系？

  • Mybatis 优点：原生 SQL（XML 语法），直观，对 DBA 友好
  • Hibernate 优点：简单场景不用写 SQL（HQL、Cretiria、SQL）
  • Mybatis 缺点：繁琐，可以用 MyBatis-generator、MyBatis-Plus 之类的插件
  • Hibernate 缺点：对 DBA 不友好

考虑为什么大公司都用 MyBatis？

在功能上 MyBatis 和 Hibernate 没有任何区别，完全对等。从计算机技术层面来说，他们都可以归类为ORM。

最核心的原因在于，MyBatis 对 DBA 友好，SQL 可控。

比如说淘宝，在现在的系统上开发一个新项目，或者对老的代码加一个新功能，改了些代码，加了些代码。那么这里面的东西可以分成三块：

  1. 静态资源和配置：可以不动现在的代码，可以直接打包分发到 CDN 或者配置中心上，系统都不用重新部署和启动
  2. 动了Java相关代码：需要重新打包，部署，启动。代码会有人 review，比如 Leader、架构师等。review 完点同意才可以 merge，才能上线
  3. 动了 SQL 或者加了新 SQL：开发人员不知道改了的东西性能好不好，到底有没有问题，会不会产生性能崩掉，或者有没有命中索引（SQL上线是否需要加额外的索引），这种只有 DBA 知道。所以动了 SQL 需要 DBA 去 code review，然后审核通过了，上线也是需要 DBA 加索引之类的。

SQL 的风险审计，性能评估等，用 MyBatis 可以知道这一点，因为SQL 都在 Mapper文件里；而用Hibernate很麻烦，Hibernate只有在运行完，这段代码被调用到了才翻译成SQL，这时才进行相关控制已经晚了。

MyBatis 便于 SQL 优化，Hibernate 比较难。因为如果一个 SQL 写的特别复杂 ，关联了一大堆的表，Hibernate 生成出来的SQL 完全没法看了，表名和字段名都被改成了 xxx_0、xxx_1 等特别长，用关联关系生成的SQL 也不知道会长什么样。就是没办法预测用HQL生成的SQL，而 MyBatis 在 Mapper 文件里写的 SQL 是怎样，运行的时候就是怎样

— 🎯总结：就是当写复杂的HQL的时候，没办法预测它生成的SQL，不便于DBA进行SQL的风险审计和性能评估。

Spring 集成 ORM/JPA*

JPA

JPA 的全称是 Java Persistence API， 即 Java 持久化 API，是一套基于 ORM 的规范， 内部是由一系列的接口和抽象类构成

JPA 通过 JDK 5.0 注解描述对象-关系表映射关系 ，并将运行期的实体对象持久化到数据库中

核心 EntityManager – 实体类的管理器

Spring JDBC 与 ORM

Spring 管理事务

  • JDBC 层，数据库访问层，怎么操作事务？编程式事务管理
  • Spring 怎么做到无侵入实现事务？声明式事务管理：事务管理器+AOP

Spring 声明式事务配置参考

  • 事务的传播性：@Transactional(propagation=Propagation.REQUIRED)
  • 事务的隔离级别：@Transactional(isolation = Isolation.READ_UNCOMMITTED) 读取未提交数据(会出现脏读, 不可重复读) 基本不使用
  • 只读：@Transactional(readOnly=true) 该属性用于设置当前事务是否为只读事务，设置为 true 表示只读，false 则表示可读写，默认值为 false。
  • 事务的超时性：@Transactional(timeout=30)
  • 回滚：
    • 指定单一异常类：@Transactional(rollbackFor=RuntimeException.class)
    • 指定多个异常类：@Transactional(rollbackFor={RuntimeException.class, Exception.class})

Spring/Spring Boot 使用 ORM 的经验

  1. 本地事务（事务的设计与坑）
  2. 多数据源（配置、静态制定、动态切换）
  3. 数据库连接池配置（大小、重连，超时），数据库连接池千万不要配置的特别大，例如淘宝早期做数据库中间件，做服务化，其中一个原因是其他应用都需要访问用户数据库，因为都需要获取用户信息。每个应用给数据库连接池配置了100个连接，那么几十个，上百个应用，就是几千个，上万个连接，就把用户数据库的链接打满了，后面很多连接根本进不来了，卡死掉了。另外一个原因是连接池配置太大，性能反而会下降。一般情况下配几十个已经可以了，配置100个都稍微有点大了。而且数据库连接池一定要配置心跳和重连，对系统高可用非常有用。假设没有配置，数据库网断了或者down掉了，等数据库恢复的时候，这个数据库连接池是不会恢复的，需要重启应用才可以，但是如果配置了心跳和重连，那么数据库连接池就可以自己重连恢复，应用不用重启。
  4. ORM 内的复杂 SQL，级联查询。SQL 越复杂，就代表对线上数据库要求越复杂
  5. ORM 辅助工具和插件。
