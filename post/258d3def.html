Netty 原理与 API 网关

高性能

什么是高性能

  • 高并发用户（Concurrent Users）-- 从业务的角度来衡量

    可以在压测的时候模拟出某个请求的并发用户

  • 高吞吐量（Throughout）-- 从技术角度来衡量

    QPS 读请求的吞吐量

    TPS 写请求的吞吐量

  • 低延迟（Latency）-- 从技术角度来衡量

    延迟（Latency）和 响应时间（Response Time）区别：

    • 延迟： t4 - t3，针对的对象是系统；
    • 响应时间： t2 - t1，针对的对象是客户
    • 如果延迟很低，这两者的时间几乎没有区别

    压测结果里的延迟一般是指响应时间，而且是用百分数来衡量系统的延迟，因为直接使用时间来衡量有可能被平均，就像你的收入和马云的收入一平均，你的收入就过千万了。

并发量要Hold的住，系统不能崩掉 – 代表了系统的海量并发的处理能力

QPS、TPS 要高，高吞吐，低延迟 – 代表了系统很好的服务能力

高性能的副作用

  • 系统复杂度 x 10 以上
    • 可能你看到的网站的某个小页面就是一个系统，像淘宝网的收藏夹就是一个特别复杂的系统
    • 一个并发几十的系统，和一个并发几千，上万的系统，虽然都是系统，但是却是两个不同的东西。一个玩具车，一个是家用车/军用车
  • 建设与维护成本++++
    • 复杂度高了，你改一个东西，影响的范围就大了，不是你想改就能改的
  • 故障或 BUG 导致的破坏性 x10 以上
    • 出问题的时候，破坏性会很大。
    • 系统有高价值，同时意味着有很高的责任，出问题有更恐怖的后果
    • 并发很低、性能很低的系统，就像骑自行车一样，出了问题，摔一跤，爬起来接着骑；中等规模的并发的系统，就像小汽车，小汽车的维护成本就比自行车高多了，路上出了事故，人受伤；高并发系统，那就是飞机，或者战斗机，维护的成本又比小汽车高很多，飞行途中碰到时间，哪怕是碰到一只小鸟，那都会造成很恶劣的事故，有可能生还的几率很低。

应对策略

稳定性建设（混沌工程）：

混沌工程的概念是由spring cloud 那家公司提出来的

  1. 容量*

    • 首先要清楚系统到底怎么回事，压测、监控、各种指标
    • 到底能支撑多少个并发，1000个还是10000个
    • 系统的吞吐量，是能够每秒做500个订单的交易，还是能做5000个订单的交易

    如果你对现状都不了解，那么你做一个运营的活动什么的，比如说引流，引来1万，10万流量，可能洪水一下就把你的大坝冲垮了，因为你不知道你的坝有多结识。

    目前国内最高的并发，如果按交易数（TPS）来计算的话，线上实际的最高的并发大概多大？是哪家公司的？

    一天有86400秒（要记这种常见的数）。如果一个系统每天的平均并发的TPS，比如说订单是1000的话，那么每天有86400 * 1000 = 86400000，约等于一天有 8千6百万 个订单。目前国内的互联网电商，每天的订单量都没有这个量。

    第一梯队：淘宝目前大概每天有3000多万个订单

    第二梯队：像美团、京东、滴滴大概在1000万-2000万之间个订单

    也就是说淘宝每天的平均并发TPS大概在 3000万 / 86400 = 347.2 ，TPS大概是350，这个量不高；即使把时间缩短在8个小时，TPS也才大概1000；只算热点时间，像中午或晚上10点的时候，下单比较多，可能也就几千的样子。

    真正对淘宝比较有挑战的事情：

    1. 秒杀活动
    2. 双11、双12的电商活动

    对淘宝、天猫、支付宝它们的并发量最大的时候，就是比如说双11活动0点以后的第一分钟内，甚至在前几秒内，那一瞬间的并发量是最大的。把淘宝、天猫和一些外围的系统，把它们在前几分钟交易的流量全部导入到支付宝。支付宝在19年的双11的第一分钟，TPS最高的时候到达了54万，20年是58万，这是目前国内最大的并发。

    在阿里之前，国内最大的并发是改造前的12306，后来从业务改造，技术改造，把流量分流了，并发就每那么搞了，而且做了排队。

    平时的时候并发都不高的，真正高的时候是做活动时候，像秒杀。双11的时候，天猫一天大概有10亿左右的订单，是平时的几十倍。

    平时水位都不高，突然后一天水位突然很高，那一下不能把你的系统冲垮了。解决的办法上云。目前全世界三大云厂商：亚马逊，微软，阿里云，前三家云厂商，有两家都是做电商的。双11那天可能要拿比平时多5-10倍的机器，现在可能更多。但是平时不会用，这样就会有大量的机器冗余，这些冗余的资源就分给大家用。

  2. 爆炸半径

    每次去上线，改系统都有可能触发一些故障，新的功能有bug。想让一个系统少出bug，少出问题，最好的办法是永远都不动它，只要你动它就会有风险。70%左右的出故障的概率是因为上线，环境和网络变更，数据库迁移，都是这种变动引起的。既然故障是由变动引起的，那么有没有办法可以控制这些变动在某个范围内 – 爆炸半径。

    如果上线，把风险控制在比较小的范围内，比方说某个小模块，那么它出问题了，顶多这个模块受到影响。微服务将服务拆小了，也是在做爆炸半径。

  3. 工程方面积累与改进

    系统出问题有三个原因：

    天灾、人祸、所有人都没有想到的bug

Netty 如何实现高性能

Netty的特点：异步，事件驱动，NIO

回顾下事件处理机制

事件进来 -> 事件队列 -> 事件分发器 -> 事件 channel -> 事件处理器

事件处理机制 到 Reactor 模型

事件进来 -> 事件分发器（只负责接收连接，然后分发） -> 事件处理器（只负责处理）

Reactor 模式首先是事件驱动的，有一个或者多个并发输入源，有一个 ServiceHandler 和多个 EventHandlers，这个 Service Handler 会同步的将输入的请求多路复用的分发给相应的 Event Handler

Reactor 模型 到 Netty NIO

Reactor 单线程模式

非主从 Reactor 多线程模式

主从 Reactor 多线程模式

本质上还是主要分成两个部分，一个部分是处理接收请求然后分发（BossGroup），一个是负责IO的工作（WorkerGroup）

Netty 对三种模式的支持

模式 代码

serverBootstrap.group(eventGroup);

serverBootstrap.group(eventGroup);
主从 Reactor 多线程模式 EventLoopGroup bossGroup = new NioEventLoopGroup();

ServerBootstrap serverBootstrap = new ServerBootstrap();
serverBootstrap.group(bossGroup, workerGroup);

Netty 网络程序优化

粘包和拆包

ByteToMessageDecoder 提供的一些常见的实现类：

  1. FixedLengthFrameDecoder：定长协议解码器，我们可以指定固定的字节数算一个完整的报文
  2. LineBasedFrameDecoder：行分隔符解码器， 遇到 \n 或者 \r\n，则认为是一个完整的报文
  3. DelimiterBasedFrameDecoder：分隔符解码器，分隔符可以自己指定
  4. LengthFieldBasedFrameDecoder：长度编码解码器，将报文划分为报文头/报文体
  5. JsonObjectDecoder ：json 格式解码器，当检测到匹配数量的“{”、”}”或”[””]”时，则认为是一个完整的 json 对象或者 json 数组

Http 断点续传

如果没有的话，会有两个弊端：

  1. 如果下载到一半，甚至99%，然后出异常，又需要从头开始下载
  2. 没办法用多线程下载。假设是1G的文件，认为把他分成8份，每份128M。第一个请求头部range就是0-128，第二个请求头部range 128-256…

Nagle 与 TCP_NODELAY

网络拥堵与 Nagle 算法优化 TCP_NODELAY

  • MTU：Maxitum Transmission Unit 最大传输单元 1500 Byte 数据帧 – 数据链路层
  • MSS：Maxitum Segment Size 最大分段大小 1460 Byte 数据大小 – TCP 报文段

TCP_NODELAY 优化条件：

  • 缓冲区满
  • 达到超时，一般是200ms

socket 编程里面的 send 还是 receive 都没有真正的操作网卡，如果直接操作网卡的话，这个算法是没办法干的。

  • send 只是把 数据（字节数组）发送给操作系统内核，能不能发操作系统说了算
  • receive 操作系统先从网卡里接收数据到缓冲区里面，然后给我们业务

如果系统发送的都是小的包（小于1460字节，尽量卡到这个阈值，效率才是最高的），频率又低，那么按这个算法的话，缓冲区一直不满，达到200ms超时后再发的话，系统延迟就很高了。这种时候应该关了这个算法。

如果系统发送的是小的包，但是频率高（瞬间缓冲区能满的，并发高，或者对延迟不敏感），就可以使用这个算法

连接优化

TCP三次握手

TCP四次挥手

TCP 连接需要等几分钟才关闭，这时候你再用同一个端口开启TCP，会告诉你端口被占用。

有两个办法可以优化：

  1. 降低这个等待的周期，降到30s
  2. 操作系统的端口有复用参数，打开这个复用参数，在这种等待周期，又有新的TCP连接的话，就可以立刻使用该端口

典型应用：API 网关

网关的结构与功能

网关处于业务系统和客户中间，让所有的业务请求都从网关来走，但是内部请求就不需要从网关走了。就像门卫，客户来访，需要在门卫处排队登记（流量控制），验证身份。

网关的四个功能：

  1. 请求接入

    作为所有API接口服务请求的接入点

  2. 业务聚合

    作为所有后端业务服务的聚合点

  3. 中介策略

    实现安全、验证、路由、过滤、流控等策略

  4. 统一管理

    对所有API服务和策略进行统一管理

​

网关的分类

流量网关

典型应用：Nginx

使用ngx_openresty 集成到 Nginx 上就可以当 web 应用的服务器。

  • 所以早期有人这样做，将其充当应用防火墙WAF。
  • 后来又有些在上面做一些业务相关的控制。比如说交易系统的流控，就可以在上面做、运营会定期的把大客户的信息导出来，然后配到lua脚本里面去，这样就能根据客户不同的等级指定不同的流量处理，vip1的每秒可以访问100次，vip2的每秒可以访问200次…。但是这样操作难以管理，以及很不安全，因为在运营传输过程中可能泄漏数据。
  • 后来改进的版本，先在线上找一台服务器放一台redis，业务系统把用户的等级等相关信息用lua脚本放到redis里面，然后ngx_openresty 就可以通过lua脚本去redis里面拿用户的等级数据，然后再根据等级数据进行流控。

但是这样在流量网关上进行业务处理，不经济实惠且不好管理

业务网关

典型应用

Zuul

Zuul 是 Netflix 开源的 API 网关系统，它 的主要设计目标是动态路由、监控、弹性 和安全

Zuul 的内部原理可以简单看做是很多不同功能 filter 的集合，最主要的就是 pre、 routing、post 这三种过滤器，分别作用于调用业务服务 API 之前的请求处理、直接响应、调用业务服务 API 之后的响应处理

Origin Server 业务系统里面的服务器可能有多个，可以在业务网关和业务系统之间，再插入一个流量网关（Nginx），把服务器给隐藏起来

Zuul2

Zuul 2.x 是基于 Netty 内核重构的版本

核心功能：

  1. Service Discovery
  2. Load Balancing
  3. Connection Pooling
  4. Status Categories
  5. Retries
  6. Request Passport
  7. Request Attempts
  8. Origin Concurrency Protection
  9. HTTP/2
  10. Mutual TLS
  11. Proxy Protocol
  12. GZip
  13. WebSockets
